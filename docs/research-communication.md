# Research Communication Layer

A single-source-of-truth communication layer over the existing academic records. It lets the same research be presented to different audiences **without duplicating content**, and generates communication assets as **drafts for human review** — never auto-published. The peer-reviewed record (`publications.json`, `projects.json`) always remains authoritative and is never overwritten.

## Principles
- **Single source of truth.** Communication assets reference existing publication/project IDs (`pub:<anthology-id>` etc.); they never re-state or alter the scholarly metadata.
- **Nothing auto-published.** Every generated asset is labelled `DRAFT — Human review required` and lives in the private `reports/` area. Public **Spotlight** pages render only for assets whose `status` is `approved` in `data/communication.json`.
- **No fabrication.** Drafts are built from verified metadata; interpretive prose (plain-language summaries, pitches) is left as clearly-marked templates for a human to complete/verify. No research claim or result is invented or exaggerated.

## What is public (built from verified data, no fabrication)
- **Research Showcase** (`showcase.html`) — flagship publications, programmes, keynotes, patent; an overview for newcomers. Every item links to its authoritative record.
- **Research Collections** (`collections.html`) — publications auto-aggregated by existing theme tags.
- **Research Timeline** (`timeline.html`) — chronological publications + milestones from existing records.
- **Publication Spotlights** (`spotlight-<id>.html`) — generated **only** for publications with a human-`approved` plain-language summary / spotlight in `data/communication.json`. None until approved.

## What is draft-only (private, review-gated)
Generated by `python scripts/admin_sync.py --communication`:
- `reports/communication/<id>.md` — per-flagship draft set: plain-language summary (5-question framework), elevator pitches (30s/1m/3m), social drafts (LinkedIn/X/Bluesky), teaching summaries (UG/MSc/PhD + discussion questions). All prefixed `DRAFT — Human review required`.
- `reports/media-kits/<id>.md` — factual media kit (citation, author bio, ORCID, paper link, contact, profiles). Review-gated; publish only when approved.
- `reports/communication-dashboard.md` — internal readiness matrix (which assets exist / are approved per publication).

## Communication profiles
`data/communication.json.profiles` and `config/admin_sync.json.communication.profiles`: academic, industry, policy, public, students, media, funders. A per-asset `profile_variants` map holds audience-specific wording for the same underlying facts.

## How to add / approve an asset
1. Run `python scripts/admin_sync.py --communication` to (re)generate drafts into `reports/`.
2. Edit the draft; verify every claim against the paper.
3. To publish a Spotlight: add the item to `data/communication.json.assets["<pub-id>"]` with `plain_language.status: "approved"` (and optionally `spotlight` with `key_contributions`/`impact`), then rebuild. The Spotlight page appears and is linked from the publication.
4. Media kits are published only when you copy an approved kit into `downloads/` and link it.

## Visual communication (architecture-ready)
Graphical abstracts, diagrams, infographics and figures are managed like other images: drop files into `assets/source/`, run `scripts/process_images.py`, reference them from a spotlight or theme via `gallery.json`. Added gradually; the pipeline handles them consistently.

## QA
The communication layer never replaces or obscures the scholarly content: Showcase/Collections/Timeline link to authoritative records; Spotlights carry a "the paper is the authoritative source" note; drafts stay in `reports/` until approved. Build validation, accessibility and privacy checks cover all public pages.
